# Rust Bridge Update Guide

The Rust bridge is a local parser bridge only. It uses `yara-x-parser` to parse
YARA syntax and exposes a cbindgen-generated C++ ABI. C++ remains responsible for
semantic validation, lowering, scheduling, provider binding, and VM execution.

## Files

- `rust/yara_bridge/Cargo.toml` pins `yara-x-parser` and `cbindgen`.
- `rust/yara_bridge/build.rs` runs cbindgen during Cargo builds.
- `rust/yara_bridge/cbindgen.toml` controls the generated C++ header shape.
- `rust/yara_bridge/src/abi.rs` defines C ABI view structs and `BRIDGE_VERSION`.
- `rust/yara_bridge/src/ffi.rs` exports panic-safe `extern "C"` functions.
- `rust/yara_bridge/src/ast_bridge.rs` maps YARA-X AST nodes into bridge-owned
  arena data.
- `rust/yara_bridge/src/model.rs` stores the Rust-owned parsed rule arena.
- `rust/yara_bridge/src/tests.rs` covers bridge parsing and ABI accessors.
- `rust/yara_bridge/generated/rule_engine/yara_bridge.hpp` is generated by
  cbindgen and included by C++; it is a build artifact, not hand-edited source.

## ABI Rules

- Keep `ReParsedRuleSet` opaque to C++ and release it only with
  `re_yara_bridge_free`.
- Return borrowed views only while the owning `ReParsedRuleSet` is live.
- Catch panics at FFI entry points and return diagnostics instead of unwinding
  into C++.
- Add new AST fields as explicit view data. Do not expose Rust allocation
  ownership to C++.
- Bump `BRIDGE_VERSION` when the generated ABI or node meaning changes in a way
  C++ must branch on.
- Add Rust bridge tests for every newly bridged YARA-X expression shape before
  changing C++ semantic lowering.

## Updating Dependencies

From the repository root:

```powershell
cargo update --manifest-path rust\yara_bridge\Cargo.toml -p yara-x-parser --precise <version>
cargo update --manifest-path rust\yara_bridge\Cargo.toml -p cbindgen --precise <version>
```

Then update the matching version pins in `rust/yara_bridge/Cargo.toml`.

## Regenerating The Header

The CMake build invokes Cargo, and Cargo runs cbindgen through `build.rs`.
Regenerate the header with either Cargo or CMake:

```powershell
cargo build --manifest-path rust\yara_bridge\Cargo.toml
cmd.exe /c 'call "%VSINSTALLDIR%\VC\Auxiliary\Build\vcvars64.bat" && cmake --build build\codex-debug --target rule_engine_yara_bridge_build'
```

The generated header path is:

```text
rust/yara_bridge/generated/rule_engine/yara_bridge.hpp
```

Do not edit that file manually. Change `abi.rs`, `ffi.rs`, or `cbindgen.toml`
and regenerate it.

## Required Verification

Run Rust checks:

```powershell
cargo fmt --manifest-path rust\yara_bridge\Cargo.toml -- --check
cargo test --manifest-path rust\yara_bridge\Cargo.toml
cargo clippy --manifest-path rust\yara_bridge\Cargo.toml -- -D warnings
```

Run the C++ bridge and semantic tests through the normal CMake build:

```powershell
cmd.exe /c 'call "%VSINSTALLDIR%\VC\Auxiliary\Build\vcvars64.bat" && cmake --build build\codex-debug --target rule_engine_tests rule_engine_check rule_engine_server rule_engine_client'
ctest --test-dir build\codex-debug --output-on-failure
```

If a YARA-X bump introduces new AST variants, add Rust bridge coverage first,
then update the C++ converter and semantic verifier so unsupported constructs
produce explicit diagnostics instead of silent false results.
